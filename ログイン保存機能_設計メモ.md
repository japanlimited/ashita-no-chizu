# あしたの地図 ログイン・クラウド保存 設計メモ

## 1. 目的

現在のMVPは、診断結果を「その端末・そのブラウザのLocalStorage」に保存している。

次の段階では、ユーザーがログインすることで、別のスマートフォンやPCから同じ診断結果を見られるようにする。

## 2. 推奨構成

### 最初におすすめ

```text
フロントエンド:
  GitHub Pagesで公開中の静的Webアプリ

ログイン:
  Supabase Auth

保存先:
  Supabase Database / Postgres

アクセス制御:
  Row Level Security
```

理由:

- 今のHTML/CSS/JS構成を大きく壊さずに追加できる
- メールログインやGoogleログインを追加しやすい
- データベースと認証が同じサービスで扱える
- ユーザーごとに自分のデータだけ見える制御を作れる

## 3. 代替案

### Firebase

```text
Firebase Authentication
Cloud Firestore
Firebase Security Rules
```

メリット:

- Googleログインとの相性が良い
- モバイルアプリ化のときに使いやすい

注意点:

- FirestoreはドキュメントDBなので、診断履歴や分析結果の設計を最初に整理する必要がある

### Next.js + Vercel + DB

メリット:

- 将来OpenAI APIを安全にサーバー側で扱いやすい
- 本格的なWebアプリ構成にしやすい

注意点:

- 今の静的アプリより構成が大きくなる
- 初回設定の学習量が増える

## 4. この機能で実現したいこと

必須:

- 新規登録
- ログイン
- ログアウト
- ログイン中ユーザーの診断結果保存
- 別端末でログインして診断結果を表示
- データ削除
- 未ログイン時はLocalStorage保存またはゲスト利用

できれば入れたい:

- ログイン後にLocalStorageの診断結果をクラウドへ移行
- 診断履歴一覧
- 最新の診断結果をホームに表示
- 目標の編集
- 明日の一歩の完了チェック

## 7. ログイン方式の候補

### 候補A: メールリンクログイン

ユーザーがメールアドレスを入力すると、ログイン用リンクが届く。

メリット:

- パスワードを持たなくてよい
- 初心者向けにわかりやすい

注意点:

- メール受信が必要
- 迷惑メールに入る場合がある

### 候補B: Googleログイン

Googleアカウントでログインする。

メリット:

- スマホで使いやすい
- ログインが速い

注意点:

- Supabase側でGoogle OAuth設定が必要
- Google Cloud側の設定が必要になる

### 候補C: メール + パスワード

一般的なログイン方式。

メリット:

- 見慣れている

注意点:

- パスワード再設定などの説明が必要
- 若いユーザーには少し重く感じる可能性がある

## 8. 推奨する最初の実装

最初は次の順番がおすすめ。

```text
1. Supabaseプロジェクト作成
2. profiles / diagnosis_answers / diagnosis_results / user_goals テーブル作成
3. Row Level Securityを有効化
4. 自分のデータだけ読める・書けるポリシーを作成
5. メールリンクログインを実装
6. ログイン中ならクラウド保存
7. 未ログインならLocalStorage保存
8. ログイン後にLocalStorageからクラウドへ移行
9. 別端末でログインして表示確認
10. GitHub Pagesへ反映
```

## 9. セキュリティとプライバシー

必須:

- Row Level Securityを必ず有効化する
- ユーザーは自分のデータだけ読める
- ユーザーは自分のデータだけ書ける
- ユーザーは自分のデータだけ削除できる
- OpenAI APIキーはブラウザに置かない
- 本名、住所、学校名、電話番号などを入力しない案内は残す
