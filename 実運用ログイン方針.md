# あしたの地図 実運用ログイン方針

## 1. 現在起きたこと

ログインリンク送信時に次のエラーが出た。

```text
email rate limit exceeded
```

これは、短時間に認証メールを送りすぎたため、Supabase側で一時的にメール送信が制限された状態。

## 2. 今後も起こるか

起こる可能性がある。

特に以下の場合に起こりやすい。

```text
開発中に何度もログインリンクを送る
同じメールアドレスに短時間で何度も送る
複数ユーザーが短時間に登録・ログインする
Supabase標準メール送信のまま公開運用する
```

Supabase公式ドキュメントでは、Authには不正利用防止のためのレート制限があり、制限を超えるとリクエストが拒否されると説明されている。

## 3. すぐできる対策

### 3.1 アプリ側で再送を防ぐ

実装済み。

```text
ログインリンク送信後、60秒間は再送ボタンを押せない
rate limitに当たった場合、1時間の待機メッセージを出す
```

これは誤操作や連打を防ぐための対策。

ただし、Supabase側の制限そのものをなくすものではない。

## 4. 実運用向けの推奨方針

推奨は次の順番。

```text
Phase 1. Googleログインを追加する
Phase 2. メールリンクは予備として残す
Phase 3. 本格運用前に独自SMTPを設定する
```

## 5. Phase 1: Googleログインを追加する

### メリット

```text
メール送信制限に引っかかりにくい
スマホでログインしやすい
ユーザーがパスワードを覚えなくてよい
若い世代にも使いやすい
```

### 注意点

```text
Supabase側でGoogle OAuth設定が必要
Google Cloud ConsoleでOAuth Client IDを作る必要がある
```

### 推奨UI

```text
Googleでログイン
メールリンクでログイン
ゲストで使う
```

## 6. Phase 2: メールリンクは予備として残す

Googleアカウントを使いたくない人のために、メールリンクも残す。

ただし、画面には次の注意を表示する。

```text
メールが届かない場合でも、短時間に何度も押さず、少し待ってから再送してください。
```

## 7. Phase 3: 独自SMTPを設定する

ユーザーが増える前に、Supabase標準メール送信ではなく独自SMTPへ切り替える。

候補:

```text
Resend
SendGrid
Mailgun
Amazon SES
```

初心者向けには、設定が比較的わかりやすい `Resend` から検討する。

### 独自SMTPで改善できること

```text
送信上限を実運用向けに調整しやすい
送信元メールアドレスを自分のドメインにできる
メール到達率を管理しやすい
ブランドとして信頼されやすい
```

### 独自SMTPに必要なもの

```text
独自ドメイン
DNS設定
SMTPサービスのアカウント
SupabaseのSMTP設定
送信テスト
```

## 8. 今すぐの運用判断

公開テスト段階:

```text
メールリンク + 60秒再送制限でOK
rate limitに当たったら1時間ほど待つ
```

身内テスト段階:

```text
Googleログインを追加
メールリンクは予備
```

一般公開前:

```text
独自SMTPを設定
利用規約・プライバシーポリシーを用意
Secret keyを再発行
ログイン・削除・別端末同期を確認
```

## 9. 次にやること

次の優先順位で進める。

```text
1. 1時間ほど待って、メールリンクログインが通るか確認する
2. Googleログインを追加する
3. 診断結果のクラウド保存を実際に確認する
4. 別端末でログインして同じ結果が見えるか確認する
5. 独自SMTP導入を検討する
```
